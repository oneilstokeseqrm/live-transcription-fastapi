# M5.4 Bug Evidence — Cross-Service Neo4j Writer Collision

**Discovered:** 2026-05-18 during M5.3 production E2E verification (post-M5.3 deploy of commit `aa0fd23`).
**Severity:** Blocks Phase-1-email-pipeline INITIATIVE sign-off. Bounded by the existing 2-layer guard so no data corruption.
**Symptom:** `EmailPromoted` handler in eq-email-pipeline never marks `local_enrichment_completed_at`; messages cycle through SQS until DLQ.
**Status:** Investigated, root-caused, deferred to a fresh session (see `2026-05-19-m5.4-next-session-prompt.md`).

---

## TL;DR

Two services write to the **same Neo4j `Interaction` node** but use different MERGE keys:

1. **eq-structured-graph-core** (the workflow's `EnvelopeV1.email` consumer) writes **first**, identifying the node by `(tenant_id, interaction_id)`, and does NOT set `internet_message_id`.
2. **eq-email-pipeline**'s `EmailPromoted` handler (`src/pipeline/skeleton.py:104`) writes **second**, using `MERGE (i:Interaction {tenant_id, internet_message_id})`. Because the existing node has no `internet_message_id`, the MERGE doesn't find it → falls through to a CREATE branch (line 144) → trips the `(tenant_id, interaction_id)` UNIQUE constraint → handler raises → SQS redelivers → same wall every time → 5 attempts → DLQ.

**This is NOT a documented V1 limitation.** Plan §6.3 V1 limitations #2 + #5 cover single-writer partial-retry corruption. This is multi-writer MERGE-key coordination — a separate bug class. The original design intent was multi-writer safety; the implementation has a coordination gap.

## Why this surfaced now (not earlier)

This bug has been latent since M3 deployed (2026-05-17), but was unreachable in production because every prior M5.x attempt errored at **Step 6 (DBOS workflow)** before reaching **Step 5/6 emit phase**:

- M3 + M4 deploy verification: ran §10.3 through Step 5 only; no real /approve attempts.
- M5: Step 6 timed out at httpx 120s before the workflow's Step 5 emit even started.
- M5.1: ON CONFLICT bug blocked Step 4 → never reached Step 5 emit.
- M5.2: Step 6 errored at function 3 due to Bug #4 (AccountProfile shape drift) → workflow never reached Step 5 emit.
- **M5.3 (THIS session, commit `aa0fd23`):** First time the workflow reached SUCCESS (87s end-to-end). Step 5 emit fired EnvelopeV1.email to eq-structured-graph-core. Step 6 emit fired EmailPromoted to eq-email-pipeline. **First production occurrence of two writers racing for the same Interaction node.**

So M5.3's success exposed M5.4. M5.4 is not a regression from M5.3; it's a previously-unreachable bug newly reachable.

## Fix options (decide via /codex consult + /plan-eng-review in M5.4 session)

Three architectural paths. All preserve the existing 2-layer guard. All would be unblocking. The choice has different blast-radius implications.

### Option A: Fix at the consumer (eq-email-pipeline)

Change `build_skeleton` to MERGE on `(tenant_id, interaction_id)` instead of `(tenant_id, internet_message_id)`. ON MATCH, set the missing `internet_message_id` + other email-specific fields.

```cypher
MERGE (i:Interaction {tenant_id: $tenant_id, interaction_id: $interaction_id})
ON CREATE SET i.internet_message_id = $internet_message_id, i.subject = $subject, ...
ON MATCH SET i.internet_message_id = COALESCE(i.internet_message_id, $internet_message_id),
             i.subject = COALESCE(i.subject, $subject), ...
```

**Pros:**
- Touches one repo (eq-email-pipeline).
- Idempotent on retry (no constraint trips).
- Doesn't change eq-structured-graph-core's contract.
- Tolerates ANY upstream writer order — eq-email-pipeline's job becomes "fill in email-specific fields if not already set."

**Cons:**
- Loses the cross-mailbox dedup safety the original `MERGE on internet_message_id` provided (RFC 5322 Message-ID is the cross-provider dedup key per the code comment at `skeleton.py:100`). However: the workflow's Step 5 emit ALREADY fires once per interaction_id, so cross-mailbox dedup is now upstream-of-Neo4j's responsibility. May or may not be a real loss.
- The "first writer must include internet_message_id" affordance is moved away from Neo4j.

**Risk to other services:** LOW. Only eq-email-pipeline's enrichment path changes.

### Option B: Fix at the producer (eq-structured-graph-core)

Make eq-structured-graph-core include `internet_message_id` when it writes the Interaction node. Then eq-email-pipeline's existing MERGE-on-internet_message_id works as designed.

**Pros:**
- Preserves eq-email-pipeline's cross-mailbox dedup invariant.
- One repo to change (eq-structured-graph-core).

**Cons:**
- eq-structured-graph-core is a transcript + email consumer; the EnvelopeV1.email schema must already include `internet_message_id` for the producer to read it. If it doesn't, this also requires changing live-transcription-fastapi's emit step. Two-repo coordination — same class of fragility that M1+M2 had.
- Transcript emissions don't have an internet_message_id; eq-structured-graph-core would need a conditional set.
- Other downstream consumers of EnvelopeV1.email may need to handle the new field (low risk if the field is purely additive).

**Risk to other services:** MEDIUM. Touches a service that writes Interaction nodes for both transcripts AND emails.

### Option C: Shared MERGE-key contract

Define a single canonical Neo4j MERGE pattern for `Interaction` nodes and apply it consistently across BOTH writers. E.g., `MERGE on (tenant_id, interaction_id)` everywhere, with `internet_message_id` as a property both writers cooperatively set.

**Pros:**
- Architecturally cleanest.
- Eliminates the entire class of multi-writer MERGE-key drift.
- Sets up Phase 2's Neo4j MERGE-everywhere refactor cleanly.

**Cons:**
- Larger blast radius — touches both repos and any future Neo4j writer.
- Needs a documented contract somewhere central.
- More design effort.

**Risk to other services:** MEDIUM-HIGH but isolated to Phase 1 sign-off scope if scoped tightly.

### Open question for M5.4 design phase

The Railway logs show **eq-structured-graph-core already writes MENTIONS/DISCUSSED/PART_OF/etc. via its own LLM extraction.** That overlaps significantly with eq-email-pipeline's `write_flesh` step (which ALSO writes MENTIONS via OpenAI extract). Does eq-email-pipeline's `write_flesh` add ANYTHING beyond what eq-structured-graph-core already wrote?

If no → eq-email-pipeline's handler may need a shrunk responsibility: drop `build_skeleton` + `write_flesh` + `extract` entirely; just write headline + summary + Pinecone vector + thread summary + mark complete.

This is the most important design question for M5.4. /codex consult + /plan-eng-review will help decide.
