═══════════════════════════════════════════════════════════════════════
M5.4 BUG — what we discovered (the LOAD-BEARING context)
═══════════════════════════════════════════════════════════════════════

**Symptom:** The eq-email-pipeline EmailPromoted handler never marks
`emails.local_enrichment_completed_at`. Messages cycle through SQS
attempts (VT=300s, maxReceiveCount=5) without success → eventually DLQ.

**Root cause:** Cross-service Neo4j writer collision. Two services
write the same Interaction node but use different MERGE keys:

  1. **eq-structured-graph-core** (downstream consumer of the workflow's
     Step 5 EnvelopeV1.email emit) writes FIRST. Creates the Interaction
     node by (tenant_id, interaction_id) WITHOUT setting
     internet_message_id. Also writes MENTIONS/Entity, DISCUSSED/Topic,
     SENT/Contact, RECEIVED/Contact, HAPPENED_IN/CalendarWeek, PART_OF/
     Chunk, GROUPS/ConversationThread relationships in the same pass.

  2. **eq-email-pipeline** (EmailPromoted handler, Step 6 emit consumer)
     writes SECOND. Its build_skeleton (src/pipeline/skeleton.py:104)
     does `MERGE (i:Interaction {tenant_id, internet_message_id})`.
     The existing node has no internet_message_id, so MERGE doesn't
     find it. Falls through to CREATE branch (line 144). Hits the
     (tenant_id, interaction_id) UNIQUE constraint. Handler raises.
     SQS retries. Same wall every time.

**Why latent until now:** Every prior M5.x attempt errored at Step 6
(workflow itself) BEFORE the workflow's Step 5 emit fired. M5.3 was
the first production occurrence of two writers racing for the same
Interaction node. This bug has existed since M3 deployed (2026-05-17)
but was unreachable.

**Why this is NOT a new V1 limitation:**
- V1 limitation #2 (Neo4j build_skeleton/write_flesh partial-retry
  corruption — bounded by 2-layer guard) covers a SINGLE writer's
  partial commits being retried.
- V1 limitation #5 (build_skeleton CREATE-fallback for missing
  internet_message_id) covers emails that come in without RFC 5322
  Message-ID headers.
- M5.4 is a MULTI-WRITER coordination problem — different bug class.
  The 2-layer guard prevents data corruption (no duplicate writes
  succeed), but completion is impossible.

**Bounded behavior:** No data corruption (the constraint prevents it).
Handler retries are bounded by SQS maxReceiveCount=5 → DLQ. The 2-layer
guard works as designed.

**Critical context from the user (this is the original design intent
the M5.4 fix must honor):** "we were trying to build so that we
wouldn't run into any other issues if one of the other repos wrote
first." The design INTENT was multi-writer safety. M5.4 is the
implementation closing the coordination gap.
