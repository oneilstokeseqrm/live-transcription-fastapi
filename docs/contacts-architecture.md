# Contacts Architecture

**Last updated**: 2026-04-01
**Status**: Fully operational — all services deployed and processing contact data

This document describes how contacts work across the entire microservice architecture: how they're created, persisted, enriched, and consumed. It's intended as a reference for any agent or developer working with contact data in any service.

---

## 1. What Is a Contact?

A contact is a **person** who interacts with the system — an email sender/recipient, a meeting attendee, a calendar invitee. Every contact is identified by a canonical **UUIDv4 `contact_id`** generated by Postgres. This ID is the single source of identity everywhere — Postgres, Neo4j, EventBridge envelopes, and all downstream services.

**Core principle:** Names and emails are metadata attached to the contact_id. The contact_id IS the identity. A contact can exist with only an email (name unknown) but NEVER without a contact_id.

#### FK Chain (Critical Gotcha)

The FK chain for contact links is **3 levels deep** with a **misleading column name**:

```
raw_interactions.interaction_id       ← must exist first
  → interaction_summaries.interaction_id  (FK to raw_interactions)
    → interaction_contact_links.interaction_id  (FK to interaction_summaries.summary_id)
```

**The trap:** `interaction_contact_links.interaction_id` does NOT hold a raw interaction_id — it holds an `interaction_summaries.summary_id`. The Prisma schema names this field `interactionSummaryId` but maps it to the column `interaction_id`.

When persisting contact links, you must:
1. INSERT into `raw_interactions` first (`ON CONFLICT DO NOTHING` — safe for concurrent writers)
2. INSERT into `interaction_summaries` with a new `summary_id` (both `interaction_type` and `summary_type` are NOT NULL with no default)
3. Use that `summary_id` as the value for `interaction_contact_links.interaction_id`

### 2.3 How They Stay in Sync

```
Postgres contacts.id  ←──── single source of truth for identity
        │
        │  (same UUID flows through EnvelopeV1 payloads)
        ▼
Neo4j Contact.contact_id  ←── MERGE key: (tenant_id, contact_id)
```

- Postgres owns **creation** — contacts are find-or-created by `(tenant_id, email)`.
- The canonical `contact_id` UUID flows through EventBridge envelopes to all downstream services.
- Neo4j receives the UUID and MERGEs by `(tenant_id, contact_id)` — so there's exactly one Contact node per Postgres contact.
- Multiple services MERGE the same Contact node concurrently (MERGE-everywhere pattern) — this is idempotent and race-safe.

## 3. Contact Creation

Contacts are created in Postgres by exactly **two services**. No other service creates contacts.

### 3.1 eq-email-pipeline (email ingestion)

**Pattern:** `find_or_create_contact()` in `src/persistence/postgres.py:72-135`
- Triggered when processing emails and calendar events
- Looks up contact by `(tenant_id, LOWER(email))`
- If found: fills NULL name fields only (never overwrites existing data)
- If not found: generates UUIDv4, sets `source='email_pipeline'`, `validation_status='pending'`
- Returns the canonical contact UUID

### 3.2 live-transcription-fastapi (transcript enrichment)

**Pattern:** `TranscriptEnrichmentService.enrich()` in `services/transcript_enrichment.py`
- Triggered when processing transcripts (before LLM cleaning)
- Matches transcript timestamp against `calendar_events` using a time window (default: start-5min to end+15min)
- For each attendee from the matched calendar event:
  - Queries Postgres `contacts` by `(tenant_id, email)`
  - If found: uses existing contact_id
  - If not found: creates new contact with `source='transcript_enrichment'`
- Optionally uses Tavily API for name resolution on email-only contacts (3s timeout, max 5 lookups)
- Returns `EnrichmentResult` with contacts, contact_ids, meeting_title, front-matter

**Name resolution heuristic** (when calendar has email but no display_name):
- `jane.smith@company.com` → "Jane Smith" (split on `.`, `-`, `_`, title-case)
- `jsmith@company.com` → not confident enough → try Tavily → fall back to email-only
- `noreply@company.com` → skip entirely (generic/service email pattern)

### 3.3 What Downstream Services Do NOT Do

Downstream services (eq-structured-graph-core, action-item-graph, opportunity-forecasting) **never create contacts in Postgres**. They receive already-resolved contact_ids via EventBridge envelopes and use them to create/update graph relationships in Neo4j.

### 3.4 Three-State Per-Attendee Branching (Phase 1, since 2026-05-13)

Phase 1 of the Contact Quality Initiative tightened the contact-creation contract end-to-end. Every contact-creation path (transcript pipeline + email pipeline) now applies three-state branching on the attendee/correspondent's email domain:

| Domain class | Behavior | Where the data lives |
|---|---|---|
| **PERSONAL** (gmail.com, outlook.com, etc.) | Skip entirely. No contact, no signal, no interaction link. | Nowhere — these are filtered out at ingestion. |
| **INTERNAL** (matches tenant's connected `provider_connections`, excluding public domains) | Skip queue path. Phase 2 wires internal-user contacts; Phase 1 treats as no-op. | Nowhere (today) — Phase 2 territory. |
| **BUSINESS + known account** | Find-or-create contact with `account_id=<looked-up>`; create `interaction_contact_links` row. | `contacts` + `interaction_contact_links` |
| **BUSINESS + unknown account** | NO contact, NO link. Insert/upsert into `pending_account_mappings` + insert into `pending_account_mapping_signals` capturing email/display_name/role. | `pending_account_mappings` + `pending_account_mapping_signals` (queue tables) |

**NEVER falls back to anchor account** when domain lookup misses. This was the bug Codex flagged in the 2026-05-12 design review (fallback-to-anchor preserved misattribution).

**Backend rejection of missing account_id** at every ingestion path:
- WebSocket `/listen`: closes upgrade with code 1008 if `X-Account-ID` header is absent.
- `/text/clean`, `/batch/process`, `/upload/init`: return 400 (auth-context layer) if header is absent, or 422 (Pydantic layer) if body field is missing — depending on which validation fires first.

**Owner determination** (locked 2026-05-13):
- Transcript signals: the recording user (authenticated `pg_user_id` on the request).
- Email signals: the user whose `provider_connection` sent/received the email.
- Calendar signals: the user whose `provider_connection` surfaced the calendar event.
- First-owner-wins under concurrency: subsequent UPSERTs on `(tenant_id, domain)` never reassign owner.

**Queue lifecycle Phase 1.5** (deferred):
- Worker processes `status='approved'` entries via `eq-agent-action-core POST /api/enrich`.
- Outbox-backed durability (`account_provisioning_outbox`) prevents Postgres/Neo4j divergence.
- Queue UI exposes Approve / Map / Ignore actions to the owner.

**Reference:** `docs/superpowers/specs/2026-05-12-contact-quality-initiative-design.md` (canonical design doc).

## 4. Contact Flow Through the Pipeline

### 4.1 Full Data Flow

```
Audio/Text Input
       │
       ▼
   Transcribe (Deepgram)
       │
       ▼
┌──────────────────────────────────────┐
│  TranscriptEnrichmentService         │
│  (live-transcription-fastapi)        │
│                                      │
│  1. Match calendar event             │
│  2. Resolve attendees → contacts     │
│  3. Build YAML front-matter          │
│  4. Return EnrichmentResult          │
└──────────────┬───────────────────────┘
               │
               ▼
   front_matter + transcript → GPT-4o cleaning
               │
      ┌────────┴─────────┐
      │                  │
      ▼                  ▼
   Lane 1              Lane 2
   EnvelopeV1          IntelligenceService
   → EventBridge       → Postgres:
   extras:               raw_interactions
     contact_ids          interaction_summaries
     contacts[]           interaction_contact_links
     meeting_title        calendar_event_interaction_links
     calendar_event_id
      │
      ▼
   EventBridge → SQS (concurrent, no ordering)
      │
      ├──► eq-structured-graph-core
      │    → MERGE Contact nodes (contact_id, email, name, role)
      │    → MERGE ATTENDED/SENT/RECEIVED/CREATED
      │    → MERGE WORKS_FOR, ENGAGED_ON
      │
      ├──► action-item-graph
      │    → Contact names in LLM extraction prompts
      │    → Seed owner resolver with known contacts
      │    → MERGE ENGAGED_ON (Contact→Deal)
      │
      ├──► opportunity-forecasting (reads only)
      │    → Contact→ENGAGED_ON→Deal (1 hop)
      │    → Contact→ATTENDED→Interaction (engagement counts)
      │
      └──► thematic-lm (no contact usage)
```

### 4.2 EnvelopeV1 Payload Contract

The enriched envelope carries contact data in `extras`:

```python
extras = {
    # Existing fields (unchanged)
    "user_name": "Pete O'Neil",
    "platform": "macos",

    # Contact enrichment fields
    "contact_ids": [                # List of UUIDv4 strings
        "a1b2c3d4-...",
        "e5f6g7h8-...",
    ],
    "contacts": [                   # Full metadata per contact
        {
            "contact_id": "a1b2c3d4-...",   # UUIDv4 — ALWAYS present
            "email": "jane@acme.com",        # ALWAYS present
            "name": "Jane Smith",            # May be null
            "role": "organizer"              # organizer | attendee | optional
        }
    ],
    "meeting_title": "Q3 Pipeline Review",
    "calendar_event_id": "uuid-of-matched-calendar-event",
    "enrichment_source": "calendar_match",  # or "none"
    "enrichment_confidence": "high"         # high | medium | none
}
```

**Contract rules:**
- `contacts[].contact_id` — ALWAYS a UUIDv4 from Postgres
- `contacts[].email` — ALWAYS present
- `contacts[].name` — MAY be null (email-only contacts where name resolution failed)
- `contact_ids` is always `[c.contact_id for c in contacts]` (redundant for convenience)
- `content.text` includes YAML front-matter at the top (if enrichment succeeded)
- If no calendar match: `contact_ids=[]`, `contacts=[]`, no front-matter

### 4.3 YAML Front-Matter (in transcript text)

When enrichment succeeds, a YAML header is prepended to the transcript before LLM cleaning:

```yaml
---
type: meeting
title: "Q3 Pipeline Review"
date: 2026-03-12T14:00:00Z
attendees:
  - jane@acme.com (Jane Smith) [organizer]
  - bob@acme.com (Bob Jones)
---

SPEAKER_0: Let's get started...
```

This gives GPT-4o attendee context for speaker attribution. It's a complementary signal — structured contact metadata also travels via `extras`.

## 6. Key Architectural Decisions

### 6.1 MERGE Key Standardization

**Decision:** All Neo4j Contact MERGE operations use `(tenant_id, contact_id)`.

**Problem it solved:** eq-email-pipeline was MERGEing by `(tenant_id, email_address)` while eq-structured-graph-core was MERGEing by `(tenant_id, contact_id)`. Same person → two different Contact nodes.

**Why contact_id wins:**
- It's the Postgres-canonical identity (source of truth)
- Every contact always has a contact_id (generated on creation)
- Emails can change; the UUID is permanent
- Both pipelines resolve contacts in Postgres first — they always have the UUID

### 6.2 MERGE-Everywhere Pattern

**Principle:** Any service that has the data to create a node or relationship should MERGE it.

Three independent consumers process the same EventBridge envelope concurrently — there is no guaranteed execution order. MERGE is idempotent: first writer creates, subsequent writers match the existing node.

**Applied to:** Interaction nodes, Contact nodes, Contact→Interaction relationships, Contact→Deal relationships.

### 6.3 Unconditional SET with COALESCE

**Problem:** Multiple services MERGE the same node but use `ON CREATE SET` for different properties. Whichever runs first "wins" and the other service's properties are never written.

**Fix:**
```cypher
-- WRONG: lost if another service created the node first
MERGE (i:Interaction ...) ON CREATE SET i.title = $title

-- CORRECT: always applied regardless of race
MERGE (i:Interaction ...) SET i.title = COALESCE(i.title, $title)
```

### 6.4 Tenant Isolation

**CRITICAL:** Every database query MUST include `tenant_id`. Never perform cross-tenant queries, lookups, or data sharing. If data doesn't exist within the tenant's scope, it doesn't exist.

## 8. Known Gotchas

### 8.1 Source Field Validation

Downstream consumers validate the `source` field on EnvelopeV1 against a strict Pydantic Literal enum:
```
zoom | gmail | generic | web-mic | upload | api | import | email-pipeline | outlook
```

live-transcription-fastapi does NOT validate this field. If an invalid value reaches downstream, the envelope is **silently rejected** (logged as error, message acknowledged and discarded — does NOT go to DLQ).

**Rule:** Always use `source='api'` for test submissions via `/text/clean`.

### 8.2 FK Chain Column Naming

`interaction_contact_links.interaction_id` actually holds `interaction_summaries.summary_id`, not the raw interaction_id. The Prisma schema names it `interactionSummaryId`. See section 2.1 for the full chain.

### 8.3 NOT NULL Columns Without Defaults

Both `raw_interactions.interaction_type` and `interaction_summaries.summary_type` are NOT NULL with no default. Must be explicitly provided (typically `'meeting'`).

### 8.4 Silent Error Swallowing

`_persist_contact_links` in `intelligence_service.py` catches all exceptions as "non-fatal" (logged but doesn't fail the request). `asyncio.gather(return_exceptions=True)` in the router swallows both Lane 1 and Lane 2 failures. Check Railway logs if contact links aren't appearing.

### 8.5 Calendar Event Time Windows

The enrichment service matches by time window (default: start-5min to end+15min). If testing manually, make sure the calendar event's `start_time`/`end_time` bracket the current timestamp.
